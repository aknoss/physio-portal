## Current state

This is a **greenfield** project. The only file in the repo is `plan.md`, the authoritative design document. No `package.json`, no source, no migrations have been written yet — Stage 0 (monorepo foundation) has not been started.

Before doing any work, **read `plan.md`**: it carries the complete stack, schema, REST surface, DI contract, testing rules, and the staged roadmap. Treat it as the source of truth; if a user request conflicts with `plan.md`, surface the conflict before implementing.

## Target shape

PT-BR single-user web app for one physiotherapist to manage patients, recurring sessions, weekly/monthly billing, and signed monthly PDF reports. Runs locally via `docker-compose up`.

npm-workspaces monorepo (no Turborepo/Nx):

```
apps/api               Express + pg + tsyringe DI
apps/web               React + Vite + TS + React Query + Tailwind
packages/contracts     Zod schemas shared by api + web (single source of truth for types)
```

## Stack invariants

- **No ORM.** Backend uses the `pg` driver directly with parameterized queries. Repositories return TS entities by explicitly mapping `snake_case` rows to `camelCase` objects.
- **Migrations are manual SQL** in `apps/api/src/db/migrations/`, written as paired `NNNN_<slug>.up.sql` / `.down.sql`. A custom runner in `apps/api/src/db/migrate.ts` tracks applied migrations in a `_migrations` table and runs each migration inside a single transaction. **Never edit a migration that has been applied — always create a new one.**
- **IDs are `uuid`** generated by Postgres (`gen_random_uuid()` from `pgcrypto`).
- **Money is integer cents** (`price_cents`, `session_price_cents`). Never floats.
- **Session status is a native Postgres enum** `session_status`: `SCHEDULED | REALIZADA | FALTA | REMARCADA`. **Only `REALIZADA` bills** — `FALTA` / `REMARCADA` never count toward totals. The unique constraint on `(patient_id, date)` makes session generation idempotent.
- **DI** via tsyringe with composition root in `apps/api/src/container.ts`. Each module is `XRepository` (interface) + `PgXRepository` (impl, receives `pg.Pool`) + `XService` (logic) + `XController`. Abstract `Clock`, `PdfRenderer`, `FileStorage` so tests do not mock infrastructure.
- **PDF uses `pdfkit`**, not Chromium/Puppeteer — keeps the Docker image small.
- **Single-user auth:** one physiotherapist seeded from env; JWT login; every route except `POST /auth/login` is behind the auth middleware.
- **No frontend i18n layer** — PT-BR strings live directly in JSX (project is monolingual).
- **No E2E suite** — Playwright is intentionally out of scope. Smoke is a manual checklist in the README.

## Naming and contracts

- TypeScript: `camelCase` everywhere. SQL: `snake_case` columns and table names. The mapping is explicit in repositories (`row.full_name → entity.fullName`), not via a global convention.
- Zod schemas in `packages/contracts/src/*` are the single source of truth for request/response shapes. Backend validates with a `validate(schema)` middleware; frontend derives types via `z.infer<typeof schema>`.

## Testing rules (non-negotiable)

- **TDD is the default workflow.** Write the failing test first, watch it fail, then implement the minimum code to make it pass, then refactor. This applies to every behavior change — repositories, services, controllers, hooks, components, utils. Never push implementation code without a test that fails before it and passes after.
- **Coverage gates** (lines, branches, functions, statements): **100%** on `apps/api`, **80%** on `apps/web`. CI fails below those thresholds.
- Backend: Vitest + supertest. Repository tests run against a **real Postgres via Testcontainers** — never mock `pg.Pool` / `pg.Client`. Services are tested with in-memory fakes that implement the repository interfaces.
- The Testcontainer setup must round-trip migrations: apply every `up.sql`, then verify each `down.sql` reverts cleanly (up → down → up).
- Frontend: Vitest + @testing-library/react + MSW for HTTP interception.
- Coverage exclusions: `server.ts`, `db/migrations/*.sql`, type-only files.

## Planned commands (none exist yet — wired up in Stage 0+)

These scripts will land as stages are implemented. Until then the repo is not runnable.

| Command                                     | Purpose                                                    |
| ------------------------------------------- | ---------------------------------------------------------- |
| `npm run dev` (root)                        | Start API (`:3000`) and Web (`:5173`)                      |
| `npm test` (root)                           | Run backend + frontend test suites (100% gate on api, 80% on web) |
| `npm run lint` / `npm run typecheck` (root) | Static checks across workspaces                            |
| `npm run db:migrate -w apps/api`            | Apply pending migrations                                   |
| `npm run db:migrate:down -w apps/api`       | Revert the last applied migration                          |
| `npm run db:migrate:status -w apps/api`     | Show applied vs pending migrations                         |
| `npm run db:seed -w apps/api`               | Seed the physiotherapist user from env                     |
| `docker-compose up`                         | Boot postgres + api + web for local use                    |

Once Vitest is wired, run a single test with `npx vitest run path/to/file.test.ts` (add `-t "name"` to filter by title).

## Working order

`plan.md` defines **Stage 0 … Stage 10** as the implementation roadmap. Each stage is independently mergeable, leaves the app usable up to its scope, and must end at the configured coverage gates (100% api / 80% web) before moving on. When asked to implement work, identify which stage it belongs to and confirm prerequisite stages are done.

Rough grouping:

- **Stages 0–1:** monorepo bootstrap, migration runner, auth, `0001_init` migration creating all tables (`users`, `patients`, `schedules`, `sessions` + `session_status` enum).
- **Stages 2–5:** API verticals — patients → schedule + session generation → status + billing aggregates → monthly PDF.
- **Stages 6–9:** Frontend verticals mirroring the API order.
- **Stage 10:** Dockerfiles, docker-compose finalization, manual smoke checklist.
